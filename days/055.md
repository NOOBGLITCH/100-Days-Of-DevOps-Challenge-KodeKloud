# Kubernetes Sidecar Containers – Complete Solution

## Problem Statement

A web server container is running the **nginx** image. The access and error logs generated by the web server are not critical enough to be stored on a persistent volume. However, the Nautilus developers need access to the last 24 hours of logs for troubleshooting and debugging.

To achieve this while following the **Sidecar Pattern**, deploy a second container that continuously reads the Nginx access and error logs and ships them to a log aggregation service. Since both containers run in the same Pod, they can share an **emptyDir** volume for log storage.

### Requirements

* Create a Pod named **`webserver`**.
* Create an **emptyDir** volume named **`shared-logs`**.
* Create two containers:

  * **nginx-container** using the image **`nginx:latest`**
  * **sidecar-container** using the image **`ubuntu:latest`**
* Configure the sidecar container to run the following command:

```sh
sh -c "while true; do cat /var/log/nginx/access.log /var/log/nginx/error.log; sleep 30; done"
```

* Mount the **`shared-logs`** volume in **both** containers at **`/var/log/nginx`**.
* Ensure both containers are in the **Running** state.

> **Note:** The `kubectl` utility on the `jump_host` is already configured to communicate with the Kubernetes cluster.

---

# Step 1: Create the Pod Manifest

Create a file named **`webserver.yaml`**.
using vi webserver.yaml
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: webserver
spec:
  volumes:
    - name: shared-logs
      emptyDir: {}
  containers:
    - name: nginx-container
      image: nginx:latest
      volumeMounts:
        - name: shared-logs
          mountPath: /var/log/nginx
  initContainers:
    - name: sidecar-container
      image: ubuntu:latest
      command:
        - sh
        - -c
        - while true; do cat /var/log/nginx/access.log /var/log/nginx/error.log; sleep 30; done
      restartPolicy: Always
      volumeMounts:
        - name: shared-logs
          mountPath: /var/log/nginx
```

```bash
kubectl apply -f webserver.yaml
```

# 💡 Good to Know

## **Sidecar Pattern**

* **Purpose:** Runs a helper container alongside the main application container.
* **Separation of Concerns:** Each container has a single responsibility.
* **Shared Resources:** Containers in the same Pod share the network, storage, and lifecycle.
* **Common Use Cases:** Logging, monitoring, proxying, and data synchronization.

## **Log Aggregation**

* **Log Shipping:** Sidecar containers collect and forward logs.
* **Centralized Logging:** Logs can be sent to a centralized logging solution.
* **Log Processing:** Logs can be parsed, filtered, or transformed before forwarding.
* **Multiple Sources:** A sidecar can collect logs from multiple files.

## **Sidecar Benefits**

* **Modularity:** Separates auxiliary tasks from the main application.
* **Reusability:** The same sidecar can be reused with different applications.
* **Independent Updates:** Update the sidecar without changing the application container.
* **Resource Isolation:** Configure separate CPU and memory requests/limits.

## **Implementation Patterns**

* **Shared Volumes:** Use an `emptyDir` volume for sharing files.
* **Network Sharing:** Containers communicate over `localhost`.
* **Lifecycle Management:** All containers start and stop together.
* **Health Monitoring:** Monitor both the application and the sidecar.

## **Best Practices**

* Set CPU and memory resource requests/limits.
* Configure readiness and liveness probes.
* Handle failures gracefully in the sidecar.
* Rotate logs to avoid filling the node's disk.
* Monitor both the application and helper containers.

## **Common Sidecar Use Cases**

1. Logging sidecar – Ship application logs to a centralized logging system.
2. Monitoring sidecar – Collect and export metrics.
3. Proxy sidecar – Manage network traffic or TLS termination.
4. Data synchronization sidecar – Synchronize files or data between systems.
5. Configuration sidecar – Reload configuration without restarting the application.

> **Note:** The original solution incorrectly used `sidecar-container` as an `initContainer`. A sidecar **must** be defined under the `containers` section so that it runs alongside the `nginx-container`.
