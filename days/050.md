# Create a Pod with Resource Requests and Limits

The Nautilus DevOps team wanted to deploy an Apache HTTP Server Pod with specific CPU and memory resource requests and limits. We created the Pod using a Kubernetes manifest and verified the resource configuration.

## Requirements

* Create a Pod named **`httpd-pod`**
* Container name: **`httpd-container`**
* Image: **`httpd:latest`**
* Configure resource requests:

  * Memory: **15Mi**
  * CPU: **100m**
* Configure resource limits:

  * Memory: **20Mi**
  * CPU: **100m**

---

# Steps We Performed

## 1. Verify Cluster Connectivity

```bash
kubectl get nodes
```

---

## 2. Create the Pod Manifest

```bash
vi httpd-pod.yaml
```

Paste the following:

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: httpd-pod
spec:
  containers:
    - name: httpd-container
      image: httpd:latest
      resources:
        requests:
          memory: "15Mi"
          cpu: "100m"
        limits:
          memory: "20Mi"
          cpu: "100m"
```

Save and exit.

---

## 3. Deploy the Pod

```bash
kubectl apply -f httpd-pod.yaml
```

Expected:

```text
pod/httpd-pod created
```

---

## 4. Verify the Pod

```bash
kubectl get pods
```

Expected:

```text
NAME        READY   STATUS    RESTARTS   AGE
httpd-pod   1/1     Running   0          10s
```

---

## 5. Verify Resource Requests and Limits

```bash
kubectl describe pod httpd-pod
```

Look for:

```text
Limits:
  cpu:     100m
  memory:  20Mi

Requests:
  cpu:     100m
  memory:  15Mi
```

---

# Good to Know?

### Resource Requests

* Minimum resources guaranteed to a container.
* Used by the scheduler when placing Pods on nodes.
* Prevents overcommitting critical workloads.

### Resource Limits

* Maximum resources a container can consume.
* Prevents a container from using excessive CPU or memory.
* Exceeding the memory limit can result in an **OOMKilled** event.

### CPU Units

* `100m` = **0.1 CPU core**.
* `1000m` = **1 CPU core**.

### Memory Units

* `Mi` = Mebibytes (base 2).
* `Gi` = Gibibytes.

### Requests vs Limits

* **Requests** determine scheduling.
* **Limits** restrict runtime resource usage.
* Requests should generally be less than or equal to limits.
